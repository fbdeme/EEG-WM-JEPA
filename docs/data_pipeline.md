# Data Pipeline Design

> 최종 업데이트: 2026-03-31
> 버전: v3.0 (REVE 전체 사전학습 + 4종 다운스트림)

## 핵심 원칙

1. **"채널 수를 인위적으로 조작하지 않는다."** — Channel Mixer가 가변 채널을 처리하는 것이 모델의 핵심 기여
2. **"최대 규모의 이질적 데이터로 사전학습한다."** — REVE 전체 61,415시간, 32종 채널 구성

## 사전학습-평가 데이터 분리 전략

### 배경

TUAB/TUSL 등은 TUH의 부분집합이며, REVE에 TUH가 포함되어 있다.
SSL 사전학습은 라벨을 사용하지 않으므로, 대부분의 논문(LuMamba, BIOT 등)이 TUH 사전학습 + TUAB 평가를 leakage로 취급하지 않는다.

### 기존 연구들의 처리 방식

| 모델 | 사전학습 데이터 | TUAB 평가 | 분리 전략 |
|---|---|---|---|
| **Laya** | TUH 전체 | ✅ | 피험자 단위 제외 (가장 엄격) |
| **LaBraM** | ~20개 (TUH 미사용) | ✅ | 데이터셋 단위 제외 |
| **LuMamba** | TUEG 전체 | ✅ | "벤치마크 제외" (피험자 단위 불명) |
| **BIOT** | TUH | ✅ | 공식 split 사용 |

**대부분의 논문이 TUH 사전학습 + TUAB 평가를 문제 없이 수행하고 있음.**

### 우리의 전략: REVE 전체 사전학습 + 4종 다운스트림

```
사전학습: REVE 전체 (61,415시간, 32종 채널, TUH 포함)
    → 라벨 미사용 SSL이므로 leakage 논쟁 최소화

다운스트림 (4종):
    ① TUAB     — 표준 벤치마크, 대부분의 baseline과 직접 비교
    ② TDBrain  — TUH와 무관, 다른 몽타주(26ch), 전이 능력 증명
    ③ APAVA    — TUH와 무관, 알츠하이머 탐지, 전이 능력 증명
    ④ BCI (MI) — TUH와 무관, 운동 상상, BCI 실용성 증명
```

비TUH 벤치마크(②③④)에서도 우수한 성능을 보이면:
- "leakage 의심 없는 환경에서도 우수" → leakage 반론 완전 차단
- "사전학습에서 본 적 없는 채널 구성에서도 작동" → Topology-Invariant 핵심 증거

## 전체 파이프라인 아키텍처

```
┌─────────────────────────────────────────────────────┐
│ Offline Preprocessing (1회성, 전처리 머신)             │
│                                                     │
│ HuggingFace (REVE 원본) → 1개 파일 다운로드            │
│   ↓                                                 │
│ ① 채널명 표준화 (Alias → 10-20, REVE Position Bank)   │
│ ② 유효 좌표 없는 채널 드롭 (드롭 비율 로깅)              │
│    → 유효 채널 50% 미만인 데이터셋은 제외               │
│ ③ 리샘플링 → 200Hz                                    │
│ ④ Bandpass → 0.5~75Hz                                │
│ ⑤ 2초 윈도우 분할 (400 samples)                        │
│   ↓                                                 │
│ HuggingFace (내 Private repo)에 업로드                 │
│   ↓                                                 │
│ 원본 .edf 삭제 (디스크에 안 쌓임)                       │
└─────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────┐
│ Online — Dataset.__getitem__ (학습 머신)              │
│                                                     │
│ HuggingFace에서 스트리밍 로드                          │
│   ↓                                                 │
│ ⑥ CAR 적용 (패딩 전이므로 마스킹 불필요)                 │
│   x = x - x.mean(dim=0, keepdim=True)               │
│   ↓                                                 │
│ 반환: { eeg: [C,400], coords: [C,3], num_channels: C }│
└─────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────┐
│ Online — Collate Function                            │
│                                                     │
│ max_c = max(배치 내 각 샘플의 num_channels)            │
│                                                     │
│ ⑦ eeg: [B, max_c, 400] (부족분 zero-padding)         │
│ ⑧ coords: [B, max_c, 3] (부족분 zero-padding)        │
│ ⑨ padding_mask: [B, max_c] (True = 패딩된 가짜 채널)  │
└─────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────┐
│ Model — Channel Mixer                                │
│                                                     │
│ Cross-Attention에서 padding_mask 적용:                │
│   패딩된 채널의 attention weight → -inf → softmax 후 0 │
│   → 가짜 채널이 Query의 정보 수집에 영향 안 줌            │
│                                                     │
│ 출력: [B, N, D] ← 채널 수와 완전히 무관                 │
└─────────────────────────────────────────────────────┘
```

## 샘플 저장 포맷

```python
{
    "eeg": np.array([C, 400]),       # 2초 윈도우 (float32)
    "coords": np.array([C, 3]),      # 전극 3D 좌표
    "channel_names": ["Fp1", ...],   # 디버깅/분석용
    "source_dataset": "tuh",         # 출처 추적 (REVE 내 원본 데이터셋명)
    "num_channels": C,               # 가변 채널 수 (int)
}
```

## 채널 Alias 처리

REVE Position Bank (543개 전극 커버)를 사용하여 매핑:
- 10-20, 10-10, 10-05 시스템
- EGI 256ch, BioSemi 128ch
- 대소문자 변형 (Cz/CZ, Fp1/FP1)

```
T3 → T7 (구→신 명명법 통일)
EEG1 → 매핑 불가 → 드롭
C3-A1 → 매핑 불가 → 드롭
```

## CAR 적용 순서 근거

| 방식 | 장점 | 단점 |
|---|---|---|
| 오프라인 적용 | 학습 시 연산 없음 | 원본 복구 불가, 다른 reference 실험 불가 |
| collate 후 on-the-fly | - | 패딩 채널이 평균에 포함되어 신호 왜곡 |
| **Dataset 내 on-the-fly** | 유연성 유지, 마스킹 불필요, 오버헤드 미미 | - |

**확정: Dataset 내 적용 (패딩 전)**

## 메모리 스파이크 대응 (필요 시)

REVE의 82%가 19~22ch이므로 대부분의 배치에서 패딩은 최소 수준.
128ch가 섞이는 경우는 드물고, 그때만 메모리가 일시적으로 증가.

만약 OOM 발생 시 (모델이 아닌 인프라 레벨에서 해결):
1. Gradient accumulation으로 실질 배치 크기 유지
2. 채널 수 기반 동적 batch_size (큰 채널 → 작은 배치)
3. 극단적으로 큰 채널(>128)만 예외적으로 제외

## 기각된 대안들

| 대안 | 기각 이유 |
|---|---|
| TUH 별도 확보 | REVE에 이미 TUH 포함 (26,847시간). 중복 데이터 발생 |
| REVE Open Subset만 사용 (TUH 제외) | 21ch/22ch 사라짐, 채널 다양성 감소, 데이터 44% 손실 |
| 피험자 단위 제거 (Laya 방식) | REVE 메타데이터에 피험자 ID 없어 구현 불가. SSL은 라벨 미사용이므로 불필요 |
| 채널 상한(64ch) + 서브샘플링 | "모든 채널 수 대응" 주장과 모순 |
| Bucket Sampling | 불필요한 복잡도, 82%가 저밀도라 버켓 불균형 |
| Mask-aware CAR | CAR을 패딩 전에 적용하면 불필요 |
